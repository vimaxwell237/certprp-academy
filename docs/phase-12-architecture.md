# Phase 12 Architecture

Phase 12 adds in-app booking lifecycle notifications and tutor follow-up tooling for live sessions.

## Scope

- In-app notifications persisted in Supabase
- Tutor session follow-up records for completed sessions
- Notification center route and unread badge in navigation
- Automatic notifications for session booking, confirmation, cancellation, reminders, completion, and follow-up creation
- Learner visibility into tutor follow-up notes
- Dashboard widgets for unread alerts, latest session update, latest follow-up, and tutor follow-up workload

## Data model

Phase 12 introduces:

- `notifications`
  - one row per in-app alert
  - targets a single user
  - stores type, title, message, optional link, unread state, and a dedupe key
- `tutor_session_followups`
  - one follow-up per tutor session
  - stores tutor ownership, learner ownership, summary, recommendations, homework, and next steps

## Integrity model

The database now enforces:

- one follow-up per session
- follow-ups only for valid completed sessions
- follow-up tutor and learner ownership must match the session
- notification reads are restricted to the owning user
- duplicate reminder/event notifications are prevented through `dedupe_key`

Lifecycle notifications for booking and follow-up events are generated by database triggers on:

- `tutor_sessions`
- `tutor_session_followups`

Reminder notifications stay app-driven and are generated opportunistically when notification/dashboard data is fetched, using dedupe keys so each reminder is only created once per user/session window.

## App feature layers

Notifications live under `src/features/notifications`:

- `data/notification-service.ts`
  - reminder generation
  - unread count fetching
  - notification list fetching
  - mark-as-read
  - dashboard notification snapshot
- `actions/notification-actions.ts`
  - mark notification read
- `components/notification-card.tsx`
  - notification UI for `/notifications`

Follow-up persistence remains in `src/features/scheduling/data/scheduling-service.ts` because it is part of the tutor session domain.

## UI model

Phase 12 adds:

- `/notifications`
- unread notification badge in the global navbar
- follow-up sections inside tutor and learner session cards
- tutor follow-up form for completed sessions
- dashboard widgets for unread notifications, latest session update, latest tutor follow-up, and tutor follow-up workload

## Phase 13 recommendation

Phase 13 should focus on outward communication and scheduling quality-of-life rather than new domains:

- email notifications for key session events
- optional reminder delivery jobs
- better tutor workload filtering and session triage
- session-detail views if the scheduling surface needs deeper drill-down
